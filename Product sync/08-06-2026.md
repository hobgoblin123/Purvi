### Discussion

**Pattern identified: "What can you do better next time?" is the highest-value question.**

Nithin has asked this question at least twice now — once during the Nithins Gallery mute-music toggle session, and again today during the finance_sheet edit sheet session. Both times, the question came after a wasted round where the AI built the wrong thing. Both times, the reflection produced a durable principle that was saved (first to memory, now to Janmam).

This is a repeating product sync pattern worth encoding into Purvi's skill set:
- The user observes a failure mode
- Asks the AI to reflect on it
- The reflection produces a reusable principle
- The principle gets captured so it compounds across future sessions

The question itself is more valuable than the fix. The fix solves one instance. The question produces a rule that prevents a class of future mistakes.

**Insight:** Janmam exists because this pattern was already happening informally via memory. Memory captures the rule — Janmam captures the rule *and* traces it back to the conversation that birthed it, so the reasoning stays visible and can be challenged or refined later.

**Principle: Ground truth check — write it down before you forget.**

Tried to create the Purvi GitHub repo. `gh` CLI wasn't authenticated. Went through installing it, asked Nithin to authenticate interactively — only to discover git credentials were already configured at `/root/github-backup/.git-credentials`. The token was right there. We just didn't know we knew.

This is operational knowledge that existed but wasn't written down in a place that surfaces when needed. So it got rediscovered instead of reused.

Two sides to this:
1. **Before setting up anything** — check what's already in place. Don't assume nothing exists.
2. **When you discover something critical** — write it down immediately. If it's not written, it's not known.

Guiding rule from Nithin's first boss: **"If it's ambiguous, don't send it."** The goal is zero ambiguity. If something could be misunderstood, unclear, or forgotten — make it explicit before moving on.

---

**Rule: Track to-do's during every sync. Review before wrapping.**

To-do's that come up mid-conversation get lost if they're not written down immediately. Every sync must end with a to-do review — what's done, what's still open, what carries forward.
